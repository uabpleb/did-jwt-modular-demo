One composite signer/verifier, three algorithms

A single CompositeSigner and a single CompositeVerifier serve EdDSA (did-jwt's built-in SoftwareSigner), EIP712 (Ethereum wallet typed-data, external module), and WebAuthn (passkey, external module) — dispatched by JWT alg. The passkey-only deep dive (presentations, replay, device binding) lives on the passkey page.

The composite pair

Algorithms currently registered on the two shared instances. Registering a module adds its algs; nothing else in the page changes.

compositeSigner.supportedAlgorithms() compositeVerifier.supportedAlgorithms()

Identities

Three independent issuers, one per algorithm module. The Ed25519 key is generated in-page on load; the EIP-712 identity is your connected MetaMask account (it signs the typed data and its did:ethr resolves through MetaMask's provider); the passkey persists in the authenticator.

EIP712 — MetaMask wallet

Sign

Every button calls the same API with the same signer instance — only alg (and the issuer DID) differ.

Verify

The shared CompositeVerifier dispatches each JWT to the module registered for its alg. The software-only button is the negative control: a bare SoftwareVerifier still verifies EdDSA but cannot handle the external algorithms. Domain policy pins the EIP-712 module to a different application domain, rejecting an otherwise-valid signature. Tamper flips one payload byte per JWT.

verifyJWT(jwt, { resolver }, compositeVerifier)
