did-jwt × WebAuthn passkey

A passkey acting as a did-jwt signer/verifier under a dedicated alg: WebAuthn, driven through the real did-jwt fork and the did-jwt-webauthn-signer module. Work through the steps — each one shows the actual API call it makes and the bytes it produces. For the multi-algorithm composite demo (EdDSA + EIP-712 + WebAuthn through one signer/verifier pair), see the unified page.

Identity

1

Register a passkey

Creates a resident ES256 (P-256) passkey. Its public key becomes a did:jwk — the issuer DID. No private key ever leaves the authenticator.

registerPasskey({ rpId, rpName, userName, requireDeviceBound })
2

Sign a VC-JWT

did-jwt builds the JWT and hands the signing input to the passkey. The authenticator prompts for touch/biometric and signs authenticatorData ‖ SHA-256(clientDataJSON) — never the JWT directly.

createJWT(payload, { issuer: didJwk, signer: new WebAuthnSigner(identity), alg: 'WebAuthn' })
3

Verify

did-jwt resolves the did:jwk to a DID document, then runs the async WebAuthnVerifier, which rebuilds the signed structure and checks the P-256 signature with WebCrypto.

verifyJWT(jwt, { resolver }, new WebAuthnVerifier(rpId, { requireDeviceBound }))
4

Present (Verifiable Presentation)

The holder wraps the VC in a W3C Verifiable Presentation with a fresh nonce and signs the VP with the passkey. Verifying it proves the holder controls the (optionally device-bound) key; the chain walk re-verifies the embedded VC.

createJWT(vpPayload, { signer, alg: 'WebAuthn' }) → verifyJWT(vp) → verifyJWT(embedded vc)
5

Replay a stale presentation

A valid signature is not enough — a presentation must be fresh. The verifier issues a new challenge nonce, but we re-submit the VP from step 4 (bound to the old nonce). The passkey signature still verifies, yet the verifier rejects it because the nonce no longer matches.

verifyJWT(oldVp, …) ✓ signature — but payload.nonce ≠ freshChallenge ✗
6

Tamper

Flip one byte in the payload segment while keeping the original signature, then re-verify. A correct verifier must reject it.

verifyJWT(tamperedJwt, …) // expected to throw
